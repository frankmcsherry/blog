# Improving `columnar` with Claude

The [`columnar`](https://github.com/frankmcsherry/columnar) crate store data in a columnar representation, and provides idiomatic Rust access (largely) while only using few-but-large backing allocations.
It has no `unsafe` code (but does rely on [`bytemuck`](https://crates.io/crates/bytemuck) to launder the unsafe type conversions).

Because the allocations are few and large, it's actually pretty cheap to live-deserialize the data.
Starting from a single `&[u64]` (not a `&[u8]`; you'll see why), there is a strongly typed decode pipeline that can produce *the appearance of* complex types that you've defined, and provide random access.
One important question then is how efficient is it to work with `&[u64]` data?
Is it possible to largely do that, rather than worry about keeping typed data around?

It semeed that this was not the case.
The [rust-serialization-benchmarks](https://github.com/djkoloski/rust_serialization_benchmark) have a `columnar` entry, from me so it is meant to be good.
It's not good.

On more complex types `columnar` takes between 50ns and 300ns to "access" an element.
These are large collections of data, so it's not just that it is scanning everything.
Something small-but-non-trivial was getting in the way.

With human brainpower we could see the shape of the problem: decoding a complicated thing and then only using some of it probably trips over the complex decoding, even if you only look at some of it.

## My own damned post

The `columnar` crate does a few things, but one is to take `&[u64]` input and read through a framing header to convert it in to a sequence of `&[u8]` slices.
These slices correspond to the backing allocations of the types columnar represents.
For example, you can see these slices in an example:
```rust
#[derive(Columnar)]
enum Group<T> {
    Solo(T),
    Team(Vec<T>),
}

fn main() {
    let mut columns: ContainerOf<Group<(String, usize)>> = Default::default();
    for index in 0 .. 1024 {
        columns.push(&Group::Team(vec![
            (format_args!("Brain{}", index), index),
            (format_args!("Brawn{}", index), index),
        ]));
    }

    // Report the fixed number of large buffers backing `columns`.
    use columnar::AsBytes;
    assert_eq!(columns.borrow().as_bytes().count(), 9);
    for (align, bytes) in columns.borrow().as_bytes() {
        println!("align: {:?}, bytes.len(): {:?}", align, bytes.len());
    }
}
```
Even though there are various string-looking things there, and `Vec` to boot, the number of allocations is nine.
The nine comes from the way we represent enums (container for each variant, plus two for variant markers and offsets), vectors, and text.

So, what does the *opposite* direction look like?
If we start with a bunch of `&[u8]` there is a corresponding `FromBytes` trait, which takes these bytes back to typed Rust objects.
```rust
    // Borrow raw bytes from `columns`, and reconstruct a borrowed `columns`.
    // In practice, we would use serialized bytes from somewhere else.
    // This local function gives type support, relating `T` to `T::Borrowed`.
    fn round_trip<'a, C: ContainerBytes>(container: &'a C) -> C::Borrowed<'a> {
        // Grab a reference to underlying bytes, as if serialized.
        let borrow = container.borrow();
        let mut bytes_iter = borrow.as_bytes().map(|(_, bytes)| bytes);
        columnar::FromBytes::from_bytes(&mut bytes_iter)
    }
```
Not a lot to see here, except that we snag the bytes with `as_bytes()` and then feed them in to `from_bytes`.
At the end, we return a "borrowed" form of the container for `C`.
This is identical on the read side, but you cannot push into it any more.

Now, we rarely actually have nine unrelated `&[u8]` that decode properly.
Usually it's more like one `&[u8]` that has to hold all the data.
This calls for a "framing protocol", and the one we use actually relies on `&[u64]` data.
We want all of our slices to be `u64` aligned, because we want to be able to treat them as up to `u64`s.
The framing looks like:
```
buffers: u64
offsets: [u64]^buffers
payloads: <the rest>
```
The idea here is that `buffers` tells us how many independent buffers we should expect to extract from the coming `&[u64]`.
Then `offsets` tell us where in `payloads` to look for those buffers.
The offsets are *byte* offsets, because we need to understand the actual number of bytes to hand off.
But the logic is to use the offsets and produce multiple `&[u64]` from `payloads`.

Great! We now have enough independent slices, and can call `FromBytes::from_bytes`.
And it all works!

## But slowly

You saw at the top of the post that decoding and then accessing a field can take 300ns.
Cool story about the zero-copy, if you just spend nearly as much time screwing around looking for the data.
Why do we spend so much time looking for the data?

This is where Claude enters the story, to help out here.

We did a few sessions trying to unpack this, with a few false starts.
Or rather, great starts and false conclusions.
We thought we found the answer and got excited, before we were certain we were correct.
I type this as I am helping Claude fix the CI issues with the "tests passing" PR that we pushed (I pushed, but Claude authored, so my fault), so this is not a new phenomenon.

The root of the problem, that accessing a thing and then not even glancing at it takes time, came down to the Rust's "deterministic error semantics".
These make a lot of sense, but to be honest I hadn't really understood before.
The Rust folks don't want the error behavior of your program to depend on the optimizer, and so even if one could prune potentially erroring evaluation because the code is dead, that will not happen.

As an easy example, consider
```rust
fn unpack(data0: &[u8], data1: &[u8]) -> (u8, u8) {
    (data0[0], data1[0])
}
```
If you call this function, but then only look at the one output, you'll still end up doing a bounds test on both arrays.
Potentially if you call this function but then look at no outputs and still end up doing the test.
I have no idea, because this is one of those skills I lack.

Anyhow, `columnar` does a fair few things that might error in decoding.
1. It accesses slices where it was told to access them by `buffers` and `offsets`.
2. It converts using `bytemuck` from `[u8]` to `[u64]` (and panicking if it fails).
3. We move the bytes along with iterators, and we could run out earlier than expected.

The problem ends up being that when decoding from a `&[u64]` and then indexing to a `(&u8, &u16, &u32, &u64)`, even if you just read the `&u32` you end up checking each of the others, because that is what decoding must do.

## Fixing things

Claude was especially helpful at proposing fixes.
My intended fix was "forking Rust", but it turns out there are better solutions.

A pattern that Claude proposed, and I'm sure other humans know but I did not, is to access the data only in ways that LLVM can confirm will not panic, and in the error cases just produce bad output.
For example, rather than access `data[lower .. upper]`, first call `.min(data.len())` on each of lower and upper, to force them in bounds.
This results in bad output, avoiding the implicit validation we were doing, but is much better than `unsafe`.
We've moved validation out of the read path, which makes a lot of sense when you say it that way.

We were heroes! Both of us! Man and Robot working together! Pew pew.

Except it didn't actually work right yet.
We didn't notice this until it was time to work on the blog post.
This is a great reason to write blog posts, btw: you actually need to write down what you think works in a way that sounds appealing to readers, and if it's not true you'll find out.
I wanted to explain how accessing the i-th field in a k-tuple was now "constant time" independent of i or k.
But it wasn't true.

The great thing that Claude helped with was actually determining whether it was true by ripping open the assembly and measuring the number of instructions.
And producing this cool chart that made it easy to read.

| Stage | k=3 (insns/branches) | k=8 (insns/branches) |
|---|---|---|
| get only (tuple already built) | 13 / 1 | 13 / 1 |
| from_bytes only (slices ready) | 81 / 10 | 186 / 25 |
| full pipeline | 131 / 14 | 271 / 29 |

Here you can see where we went wrong.
We did were focused on the `get` only, ignoring the `decode` step.
We got very excited, or at least I did, and were wrong.
Certainly I was.

But once we had these numbers we were able to chase things down some more.
There were a few other panicky moments where we were able to convince Rust/LLVM not to care by giving them clear incorrect behavior in that case (e.g. "return an empty slice if bounds are bad").
We fixed these up, but things still were not great.

The next unlock was noticing that our flow of `&[u64]` to `&[u8]` to `&[T]` for various `T` had the defect that the types lose track of the alignment.
Even if `T` is `u64`, when we go from `&[u8]` back to `&[u64]` there's a test there, and it could fail.
We know it won't fail, because we know where the bytes came from, but Rust and LLVM do not know this.

The fix was to stop trafficking in bytes at all, and use `u64` pervasively.
There are still byte-aligned types, and they need to hear about the byte-level lengths; we can't just hand a `&str` container a `&[u64]` as they might only expect seven bytes.
It's not too hard; you use `(&[u64], u8)` to provide the data and last-byte validity (I did that!).
Now we could prove to Rust/LLVM that the casts are all non-erroring.

The final unlock, at least for now, was fixing the potential iterator exhaustion.
We could try and reason about the total number of slices, but Claude had an easier fix:
```rust
    let (w, tail) = words.next().unwrap_or((&[], 0));
```
If you run out of slices, just .. hand back an empty one.
It's wrong, but maybe mid-random-access isn't the time to do validation.
And it removes the panic here, as well as the evaluation that Rust's deterministic error semantics require, even when we don't look at the results.

## Findings

With these three changes made, things were looking a lot better.
Claude was very excited about the number of instructions to access field 0 of an increasingly wide tuple, and indeed the story was great there.

| Tuple width | Old (insns/branches) | Mid (insns/branches) | Now via from_u64s (insns/branches) |
|---|---|---|---|
| k=1 | 73 / 7 | 77 / 8 | 68 / 7 |
| k=2 | 100 / 12 | 105 / 11 | 68 / 7 |
| k=3 | 126 / 17 | 133 / 14 | 68 / 7 |
| k=5 | 155 / 27 | 189 / 20 | 68 / 7 |
| k=8 | 191 / 42 | 273 / 29 | **68 / 7** |

Now, no matter how many other fields there are, we do a constant amount of work to get to where we need.
Still many instructions, because you do have to respond to those error conditions by doing something screwy, but the branches all predict the right way.

This is almost great, but the goal of accessing field i out of k in constant time was not the case.
Let's have Claude explain, because I don't want to pretend to understand any of this:
```
⏺ Fascinating. Look at eager_8_flast — the growing part is this sequence:

  cmp   x12, x9
  cinc  x12, x12, lo
  cmp   x12, x9
  cinc  x12, x12, lo
  cmp   x12, x9
  cinc  x12, x12, lo
  cmp   x12, x9
  cinc  x12, x12, lo
  cmp   x12, x9
  cinc  x12, x12, lo

  That's 5 cmp+cinc pairs for an 8-tuple (to reach field 7). Compare with
  eager_3_flast which has none of these (it reaches field 2 more directly).

  LLVM has eliminated the closure calls and the field construction, but it still
  needs to advance the iterator counter to the right position. It's computing
  "which i do I need?" by conditionally incrementing a counter — essentially
  unrolling the iterator advancement as a chain of cinc (conditional increment)
  instructions. Each skipped field costs one cmp+cinc pair (2 instructions, 0
  branches — cinc is branchless).
```
For reasons that are unclear to me, LLVM really wants to count up to the field you want to access.
I'm not sure why, and Claude reports that it is just a missing LLVM optimization.
Their loop strength reduction is reportedly based on idiom detection (reported by Claude), and this is not one of the idioms.
As the number of columns increases, in our case at 16, LLVM switches to a loop to count up, but it never gets to "maybe I should add, or max, or something else".
It is very possible there's a good reason why it cannot optimize this away, but we couldn't find it.
