# Transaction Processing as Dataflow

Databases are built on the back of "transactions": sets of commands that need to appear to happen at a single moment in time, or not at all.
Transactions can read some data, write some data, write, read, write, maybe do some other things, and then eventually end with either `COMMIT` or `ROLLBACK`, as the moment where you instruct the database either to make it real, or to just walk away.

Transactions can be pretty gory to implement because they are generally "interactive": the user doesn't express their full intent ahead of time, like in most programming languages, but reveals it to the database one command at a time.
Until the database sees a `COMMIT` everything is, and needs to be, entirely tentative.

Because things are so gory, transaction processing often lives in the "control plane" of the database.
Some number of transactions are in flight at any moment, and some careful centralized logic needs to keep track of what is real and what is not.
This often limits the effective throughput of transaction processing: the number of potentially conflicting transactions that can be resolve per second, say.

A while back I wrote about how you could do a large chunk of transaction processing [in the data plane](https://github.com/frankmcsherry/blog/blob/master/posts/2016-07-17.md), increasing the throughput and reducing the implementation complexity.
Of course, it was written in [differential dataflow](https://github.com/TimelyDataflow/differential-dataflow), so understandably folk may not have felt the implementation complexity to be that much reduced (if at all).
Today we'll do the same thing, but using SQL itself (without transactions) and [Materialize](https://materialize.com/)'s ability to incrementally maintain iterative SQL to make it all fast.

**Caveats**: This post is for educational purposes; please do not actually implement transactions this way without having a hard think about what you need.
Also, I'm pretty sure I'm not the only one to think of things this way, so there might already be real products that can do this for you by this point.
Ververica's [Streaming Ledger](https://github.com/ververica/streaming-ledger) is an example that does something similar, but I don't understand it well enough to say.
Let me know if you know better, or of other approaches!

For a final caveat, bear in mind that databases are an area where everything can make total sense, be internally consistent, and still for some reason be utterly wrong.
I am but a lowly computer scientist, and not qualified to make definitive statements about how databases work, and what they should and shouldn't do.

## A Running Example: Banks and Stuff

A classic example of non-trivial transaction processing derives from accounts containing money.

Imagine a database of users with accounts, who would like to redistribute wealth among themselves.
The participants are mercurial, and don't want to reveal their plans ahead of time, and can only react in the moment to what they see.
Likewise, the transfers should only occur if they can occur *just so*, exactly as articulated by the participant.

Imagine you get millions of these transactions a second, each reading from and writing to accounts that span multiple transactions.
There are some network round trips involved in each interactive transaction, which rules out "I'll just code hard and go real fast!"
We will need a different approach; one that builds scalability in from the beginning.

## Read Sets and Write Sets

Interactive transactions can be summed up by their *read sets* and *write sets*: 
respectively, the values that the transaction read (or failed to read), and the values that it then chose to write.
As long as those read values are in place the transaction is unambiguous about what it would do, and if they are not in place then we have absolutely no clue what it would do.

We will approach the problem of resolving large numbers of pairs of read and write sets.
"Resolving" will mean picking out a subset of pairs with the property that when put in some order, the read set of each matches the most recently previously written values.
This is a very limited way to approach transaction processing: great for our account example, but less good when your transaction reads a hash of the whole database.

Let's model intended transactions, those that end with a `COMMIT`, with a table that records for any transaction identifier its read and write sets.
```sql
-- A transaction consists values read, and values to write.
CREATE TABLE intents (id INT, is_read BOOL, k TEXT, v TEXT);
```
We'll use "keys" and "values" to describe where and what was read.
You can imagine this as a key-value store, or relations where there exists a primary key.
The ideas generalize to keyless collections, which are fundamentally key-value, where the key is the row and the value is its multiplicity.

If you introduce some rows to this table, it means you'd very much like to have your values committed, as long as the reads still hold.

## Committed .. but to Where?

Rather than move data in to and out of some durable store, our committed state will instead be just a view over `intents`.

Importantly, `intents` can change.
This has the potential to change the results of our view of what has committed, and we'll need to take some care here.
The read and write sets come from transactions that ended with a `COMMIT`, and the sets for each transaction id should not themselves change.
However, new transactions can still be added, and we need to be mindful of the identifiers they are added with.
We will end up being certain about a transaction only once we are sure that all future transactions will use a higher identifier, which is something we can force with for example epochs.

We will be able to asynchronously tidy `intents`, so that it doesn't grow without bound.
We can remove failed transactions, remove the read sets of committed transactions, and remove writes that are themselves overwritten without being observed (e.g. after we remove all those read sets).
Importantly, this work is made asynchronous, rather than on the critical path of transaction resolution.

We'll end up with an indexed view over the most recently written values, to present as if we just had the data in an index, but we'll have described the logic to derive it from `intents` declaratively (using SQL) rather than imperatively (using whatever your database is written in).

## Resolving Transactions

Each transaction can commit if the previously committed write to each key its read set matches the corresponding read value.
Let's just write that logic in SQL!
How would we do that?

It's not easy in vanilla SQL, though I'm sure you could write a massive window function with some gnarly logic.
It is pretty easy when you use recursive SQL.
Brief, at least, if "easy" doesn't sound likely.

Informally, we'll iteratively develop 
1. tentative writes (as a function of tentative commits), 
2. tentative reads (as a function of those tentantive writes), and 
3. tentative commits (as a function of the tentative reads).

It turns out we'll be better served by developing transactions that do *not* commit, those that must be rolled back, as a way of starting from the optimistic take that everything should commit until we have evidence that it shouldn't.

The following SQL does exactly that:
```sql
CREATE VIEW to_rollback AS
WITH MUTUALLY RECURSIVE 

    -- Tentative writes as a function of `rollback`.
    writes(id INT, k TEXT, v TEXT) AS (
        SELECT intents.id, k, v
        FROM intents
        WHERE intents.id NOT IN (SELECT * FROM rollback)
          AND NOT intents.is_read
    ),

    -- Reads, and the corresponding read value.
    -- These may not match, which informs `rollback`.
    reads(id INT, v TEXT, r TEXT) AS (
        SELECT id, v, (
            SELECT DISTINCT ON (k) v 
            FROM writes 
            WHERE writes.id < intents.id
              AND writes.k = intents.k
            ORDER BY k, writes.id DESC, v
        )
        FROM intents
        WHERE intents.is_read
    ),

    -- Roll back transactions with a failed read.
    rollback(id INT) AS (
        SELECT reads.id
        FROM reads
        WHERE reads.v IS DISTINCT FROM reads.r
    )

SELECT * FROM rollback;
```

If `WITH MUTUALLY RECURSIVE` (WMR) is scary jargon, it may help to talk through how to determine what it will produce as output.
Each WMR block starts with all terms in scope (`writes`, `reads`, and `rollback`), and initially empty.
It then repeatedly updates the contents of these collections by applying the stated rules, in the order stated, until their contents stop changing.
It then runs that final `SELECT` block at the bottom, because all SQL fragments need to return just one collection.

In our case we start with initially empty collections, and in particular an initially empty `rollback`.
Based on this, we'll initially imagine all writes succeed, all reads read the previously written values, and transactions must rollback if those values don't match the reads.
If any transactions must roll back, we'll need to repeat the process, as some writes may not happen now, which may change `reads` (positively or negatively), then `rollback`, etc.

Does this ever stop?
Indeed it must, but let's explain that in a different subsection.

## Convergence and Termination

The above logic always terminates, and the rate of convergence can be (but is not always) very fast.

Termination is probably easiest to grok, so let's start there.

The reads of a transaction only depend on strictly prior transactions, and so the decision to roll it back is only influenced by the state of prior transactions.
If in some iteration the smallest identifier to change in `rollback` was `i`, then in the next iteration all identifiers up through *and including* `i` will not change in `rollback`.
Each round of iteration makes permanent progress in stabilizing `rollback`, increasing the least identifier that can change by at least one.

We could take as many rounds of iteration as there are distinct identifiers in `intents`, and we can contrive inputs that will do this, but we will certainly terminate.

Convergence is more nuanced, but follows similar principles.

An transaction can only be influenced by transactions with lower identifiers *that intend writes for its read set*.
Imagine that transaction identifiers are randomly assigned (perhaps `(epoch, uuid)` rather than `int`).
If you are a transaction, and there are `k` other transactions that mean to write to your read set, there is a one out of `k + 1` chance that you have the lowest identifier.
If you have the lowest identifier you can commit, immediately and permanently!
Not just you, but all transactions that have the lowest identifier among their potential coniflcts.
Potentially a linear fraction of transactions in the first iteration alone.

One way to imagine this is as a directed graph on nodes that correspond to transactions, with directed edges from one to another if the one would write at the read set of the other.
With random identifiers sprinkled atop the nodes, the number of rounds of iterations is bounded by the longest directed path with increasing node identifiers.
Informally, it is hard to find long increasing paths, because as you travel along them the amount the values increase is a function of the number of options you had at each node.
If the graph is dense (mostly conflicts) it will happen, but if it is sparse (limited conflicts) it is substantially harder.

20-year ago me, who then was facile with random graph theory, would be mortified to read the utter helplessness of the preceding paragraph.
Let's leave it with the tl;dr that while there can be many iterations, there don't *need* to be many iterations.

## Asynchronous Maintenance

Let's talk through some of the maintenance we can perform on `intents` as we go, to avoid unbounded growth in the resources required to maintain the view.

Before we go too far, recall that the view shows us what is true at a moment in time, but with `intents` continually changing we need a wait to lock down some of the transaction identifiers.
We'll do this by imagining that the identifiers generally increase, and that we are able to guard the identifiers by some clause that ensures we are discussing a prefix of intended transactions that will not change.

One straightforward way to do this is to insist that transaction identifiers only increase, and reject insertions into `intents` if they use an identifier not strictly greater than the maximum contained therein.
Another option is to pair epochs and UUIDs as transaction identifiers, and to advance epochs whenever it feels good to resolve more transactions.
We will just use a comment to remind you that it needs to be done, and won't otherwise think about it.

One of the goals of the maintenance commands that follow is that they can be run (or not run) at any moment.
We don't need to lock down the whole system to do this maintenance.

### Removing failed transactions

Transactions that must be rolled back can simply be removed from `intents`.
They should have no impact on the result of the `to_rollback` view.
Nonetheless, their removal changes the derivation of the view, and their removal may cause non-trivial work to happen.

```sql
-- Caveat by our bounds on certain ids.
DELETE FROM intents 
WHERE intents.id IN (SELECT * FROM rollback);
```

### Removing read sets of committed transactions

A transaction is conditional on its read set lining up with the values present.
Once we know a transaction will commit, we can remove its read set and commit it unconditionally.

```sql
-- Caveat by our bounds on certain ids.
DELETE FROM intents 
WHERE intents.id NOT IN (SELECT * FROM rollback) 
  AND intents.is_read;
```

This maintenance should not change the results of `rollback`, but it may cause some work to happen.
Specifically, if the path to committing such a transaction involved false starts due to conflicting reads of tentative writes, those false starts are removed.
If this work is done after failed transactions are removed, there should be no work to correct.

If we are certain that all failed transactions have been removed, then all reads can be deleted without consulting `rollback` (which could be much faster).

### Remove overwritten writes

The two rules above remove the read sets for failed and committed transactions.
It's now possible, likely even, that some writes are no longer observed: no reads occur before a subsequent write occurs.

oThis case appears to be a bit more complicated than the previous cases, as we need to look for other commited 
```sql
-- Caveat by our bounds on certain ids.
DELETE FROM intents
WHERE NOT intents.is_read 
  AND EXISTS (SELECT FROM intents i2 
              WHERE i2.key = intents.key
                AND i2.id > intents.id 
                AND NOT i2.is_read
                AND NOT i2.id IN (SELECT id FROM rollback)) 
  AND NOT EXISTS (SELECT FROM intents i2 
                  WHERE i2.key = intents.key
                    AND i2.id > intents.id 
                    AND i2.is_read);
```

This formulation is a bit sloppy, in that it will avoid tidying up writes that have any reads, not just reads before the next committed write.
However, the context in which we imagine applying it will tidy up all sorts of reads, and should make this caveat less important.
